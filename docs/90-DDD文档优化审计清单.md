## 1. 本次优化范围

本次全量检查覆盖 `docs/` 下所有 Markdown 文档，共 49 份。所有文档均已补充 `DDD 对齐说明`，并按文档类型明确了以下内容：

- 所属限界上下文或文档层级。
- 核心聚合或模型边界。
- 数据主权和跨上下文协作方式。
- 命令、领域事件、幂等、补偿的阅读口径。
- 字段、状态、页面、流程与 DDD 模型的对应关系。

## 2. 优化规则

| 文档类型 | 优化规则 |
| --- | --- |
| 阅读指南 | 增加 DDD 阅读路径、DDD 文档层级、接口事件目录入口 |
| DDD 方法论文档 | 标明它是方法论基准，不对应单一业务上下文 |
| 业务总览 | 用核心域、支撑域、通用域、限界上下文重新解释子系统 |
| 业务流程 | 标明流程中的动作是命令，结果是领域事件，状态变化发生在聚合内部 |
| 单据与库存模型 | 标明字段服务于聚合行为、不变量、值对象、状态机和领域事件 |
| 主数据 | 标明主数据上下文是上游发布语言，强调版本、快照、事件分发 |
| 子系统设计 | 标明每个系统所属限界上下文、核心聚合、数据主权、生产/消费事件 |
| 接口与事件 | 标明接口是命令、事件是事实，必须有上下文、聚合、幂等、版本和补偿 |
| UI 设计 | 标明页面服务于命令、查询模型、聚合状态和权限边界 |

## 5. 维护规则

后续新增或修改任何供应链文档时，至少回答这 8 个问题：

1. 这份文档属于哪个限界上下文？
2. 它涉及哪些核心聚合或实体？
3. 谁拥有数据主权？
4. 哪些动作是命令？
5. 哪些结果是领域事件？
6. 哪些业务不变量必须保持？
7. 哪些异常需要补偿或人工处理？
8. 页面/列表/报表是否属于查询模型，而不是核心聚合？

## DDD 对齐说明

本文属于文档治理和设计审计层，不对应单一业务上下文。它的作用是保证所有供应链设计文档持续遵守 DDD 设计口径：先明确限界上下文，再明确聚合、不变量、命令、领域事件、数据主权和异常补偿。后续新增文档时，应回到本文更新审计状态。
