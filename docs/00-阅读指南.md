## 5. DDD 文档质量规则

新增或修改任何设计文档时，必须回答这些问题：

| 审计项 | 必须明确的内容 |
| --- | --- |
| 业务目标 | 解决什么业务问题，提升什么效率，降低什么风险 |
| 角色/场景/流程 | 谁发起，哪个系统处理，哪些状态改变，异常怎么处理 |
| 领域边界 | 属于哪个限界上下文，谁是上游，谁拥有数据主权 |
| 聚合与不变量 | 哪些聚合根、实体、值对象；哪些规则必须始终成立 |
| 命令 | 哪些动作是命令，命令由哪个应用服务编排 |
| 领域事件 | 命令成功后产生哪些事实事件，事件载荷是什么 |
| 读模型 | 页面、列表、报表是否只是查询模型，不反向污染聚合 |
| 幂等与补偿 | 幂等键是什么，重复、失败、超时、部分成功如何处理 |
| 权限与审计 | 菜单、按钮、数据范围、审批、操作日志如何覆盖 |
| 数据落库 | 表字段是否服务聚合行为、状态机、事件和追溯，而不是只服务页面 |

## 7. 后续维护规则

| 场景 | 规则 |
| --- | --- |
| 新增子系统 | 先完善业务流程和领域模型，再完善功能、数据库、接口、事件 |
| 新增字段 | 说明字段所属聚合/值对象、来源、是否可变、由什么命令改变 |
| 新增状态 | 说明初始态、终态、迁移动作、禁止动作、异常态和事件 |
| 新增接口 | 说明页面调用点、命令/查询语义、请求头、请求字段、响应字段、状态码、幂等 |
| 新增事件 | 说明触发命令、聚合变化、载荷、存储表、消费者和失败补偿 |
| 修改 TMS 相关内容 | 同时检查采购、供应商、WMS、中央库存、OMS、BMS、主数据、权限的交集 |
