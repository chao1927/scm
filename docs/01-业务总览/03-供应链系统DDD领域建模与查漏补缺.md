# 03 供应链系统 DDD 领域建模与查漏补缺

> 本文基于现有供应链系统文档，按照领域驱动设计重新审视系统边界、通用语言、子域、限界上下文、核心聚合、业务不变量、领域事件和后续缺口。它不是替代已有流程/字段/子系统文档，而是作为 DDD 视角的总索引和修正基准。

## 1. 总体结论

现有文档已经覆盖了主流程、库存流水、单据状态机、主数据字段、子系统功能和详细字段，但整体仍偏“系统功能 + 单据流程 + 字段模型”。按 DDD 视角看，还需要补强以下设计骨架：

| DDD 维度 | 当前情况 | 需要补强 |
| --- | --- | --- |
| 通用语言 | 各文档有大量术语，但缺少统一术语归属 | 建立跨上下文术语表，明确同名词在不同上下文的含义 |
| 子域划分 | 已按子系统划分，但未标注核心域/支撑域/通用域 | 明确哪些能力要重度建模，哪些只需通用实现 |
| 限界上下文 | 子系统边界基本清楚，但还未正式表达模型边界 | 明确每个上下文的模型、职责、数据主权、不负责范围 |
| 上下文映射 | 已有系统协作图和事件表，但缺少上下游关系模式 | 补充客户/供应方、防腐层、开放主机服务、发布语言等关系 |
| 聚合设计 | 分散在字段模型、状态机和详细设计中 | 抽取核心聚合总表，明确聚合根、内部对象、不变量 |
| 领域事件 | 各系统文档有生产/消费事件，但缺少统一事件目录 | 统一事件命名、来源、消费者、幂等键、事实语义 |
| 查询模型 | 页面原型和列表字段较多，但写模型/读模型未区分 | 为看板、列表、报表补充 CQRS 读模型口径 |
| 异常流程 | 已有部分差异、退货、幂等说明 | 系统化补充超收、短拣、重复回传、超时、人工介入 |

DDD 重构后的设计原则：

1. 先按业务责任划分限界上下文，再决定服务和表。
2. 每个上下文内部有自己的通用语言，不强行共享一个大模型。
3. 跨上下文只通过命令、领域事件、发布语言和防腐层协作。
4. 核心业务规则放入聚合、领域服务和状态机，不散落在页面或 SQL。
5. 库存、履约、仓内作业、调拨、售后退货作为核心域重点建模。
6. 权限、日志、字典、文件、通知作为通用域，避免过度 DDD 化。

## 2. 子域划分

### 2.1 核心域

核心域是最影响供应链效率、履约质量、库存准确性和经营成本的部分，后续详细设计要优先投入。

| 核心域 | 业务价值 | 主要上下文 | 深入建模重点 |
| --- | --- | --- | --- |
| 库存可用性 | 防超卖、支撑一盘货、减少库存差异 | 中央库存 | 库存账户、预占、扣减、释放、冻结、流水、不变量 |
| 销售履约 | 订单能否准确、快速、低成本发出 | OMS、中央库存、WMS | 分仓、预占、出库指令、履约状态、取消补偿 |
| 仓内作业 | 影响收货、上架、拣货、复核、发货效率 | WMS | 入库单、出库单、任务、容器、库位、批次、异常 |
| 调拨协同 | 平衡多仓库存，处理在途和差异 | 中央库存、WMS | 调拨单、调出、在途、调入、差异处理 |
| 逆向退货 | 决定售后体验和库存回收准确性 | OMS、WMS、中央库存 | 售后单、质检、良品/不良品、退款/换货、退供 |

### 2.2 支撑域

支撑域不一定形成核心竞争力，但对核心域稳定运行非常重要。

| 支撑域 | 主要上下文 | 设计重点 |
| --- | --- | --- |
| 采购管理 | 采购 | 采购申请、询价、采购订单、入库跟踪、退供 |
| 供应商协同 | 供应商/SRM | 准入、报价、订单确认、ASN、评分、质量协同 |
| 主数据管理 | 主数据 | SKU、供应商、仓库、货主、物流商、版本、分发 |
| 结算计费 | BMS | 费用规则、费用明细、对账、账单、发票交接 |
| 物流协同 | TMS/物流模块 | 面单、轨迹、物流商、费用、异常 |

### 2.3 通用域

通用域建议采用成熟通用模型，避免复杂领域建模。

| 通用域 | 当前系统 | 设计策略 |
| --- | --- | --- |
| 权限认证 | 权限系统 | 标准用户、角色、权限、Token、操作日志 |
| 枚举配置 | 各系统 | 允许配置标签、颜色、排序，核心状态值不随意配置 |
| 文件附件 | 各系统 | 通用附件模型，业务单据引用 |
| 消息通知 | 各系统 | 通用通知模板和发送记录 |
| 审批流 | 权限/流程能力 | 可以通用化，但审批结果必须回写业务聚合 |

## 3. 限界上下文总览

| 限界上下文 | 数据主权 | 核心模型 | 不负责什么 | DDD 关系 |
| --- | --- | --- | --- | --- |
| 主数据上下文 | 商品、供应商、客户、货主、仓库、库位、物流商 | SPU、SKU、Supplier、Warehouse、Location、Carrier | 不处理采购、销售、仓储作业 | 上游供应方、发布语言 |
| 供应商上下文 | 供应商准入、评分、协同资料 | SupplierProfile、SupplierScore、ASN、SupplierQuote | 不决定内部采购审批 | 采购的供应方，主数据下游 |
| 采购上下文 | 采购申请、采购订单、采购退货 | PurchaseRequisition、PurchaseOrder、SupplierReturn | 不执行仓库收货上架，不记库存账 | WMS/库存的上游业务来源 |
| OMS 上下文 | 销售订单、履约、售后 | SalesOrder、FulfillmentOrder、AfterSaleOrder | 不管理库位库存，不执行拣货 | 库存客户方，WMS 上游 |
| 中央库存上下文 | 可用、预占、扣减、释放、冻结、流水 | InventoryAccount、Reservation、InventoryLedger | 不执行仓内作业，不做订单经营规则 | 开放主机服务、发布语言 |
| WMS 上下文 | 仓内实物作业和库位库存 | InboundOrder、OutboundOrder、PickTask、PutawayTask、LocationInventory | 不决定销售订单是否有效，不做全局可售 | OMS/采购下游，库存事实来源 |
| BMS 上下文 | 费用、对账、账单 | BillingRule、FeeDetail、ReconciliationStatement、Bill | 不创造作业事实，只消费事实 | 多上下文事件消费者 |
| 权限上下文 | 用户、角色、权限、操作日志 | User、Role、Permission、Token、OperationLog | 不决定业务规则 | 通用域，被各系统遵奉 |

### 3.1 边界修正建议

| 主题 | 修正建议 |
| --- | --- |
| 售后上下文 | 当前售后主要归 OMS。若后续售后规则复杂，可独立为“售后上下文”，但第一版保留在 OMS 更简单 |
| TMS 上下文 | 当前详细设计未展开 TMS。若物流只是面单和轨迹，可先作为 OMS/WMS 的物流模块；若多物流商、多计费、多轨迹异常复杂，应独立 |
| 计划补货上下文 | 业务总览有计划补货，但详细设计未展开。第一版可以暂作为后续扩展，先不进入 P0 |
| 财务上下文 | BMS 只做业务计费和对账，正式收付款、发票、成本核算仍交给财务系统 |
| 库存与 WMS | 中央库存管全局可用和业务占用，WMS 管库位实物和作业事实，两个模型不能混成一套 |

## 4. 通用语言查漏补缺

### 4.1 高风险同名词

| 术语 | 容易混淆点 | 建议定义 |
| --- | --- | --- |
| 库存 | 中央库存和 WMS 都叫库存 | 中央库存叫“库存账户/可用库存/预占库存”，WMS 叫“库位库存/实物库存” |
| 出库单 | OMS、WMS 都可能有出库概念 | OMS 使用“履约单/出库指令”，WMS 使用“出库单/拣货任务” |
| 入库 | 采购入库、退货入库、调拨入库不同 | 统一为 WMS 入库作业，来源类型区分采购/售后/调拨/其他 |
| 可用 | 可销售、可预占、可拣选不是一回事 | 中央库存定义“可预占库存”，渠道定义“可销售库存”，WMS 定义“可拣选库存” |
| 冻结 | 质检冻结、库存冻结、财务冻结不同 | 必须有冻结原因、冻结范围、释放动作 |
| 货主 | 自营业务可能不明显 | 若只做自营单货主，可默认平台自身；若多客户仓储，货主必须成为库存维度 |
| 批次 | 采购批次、生产批次、库存批次不同 | WMS/库存应明确 batch_no、lot_no、production_date、expire_date |
| 完成 | 各系统完成时间不同 | OMS 完成是履约完成，WMS 完成是仓内作业完成，库存完成是流水入账完成 |

### 4.2 建议新增统一术语表字段

后续可在主数据或文档中维护术语表：

| 字段 | 说明 |
| --- | --- |
| term_code | 术语编码 |
| term_name | 中文名称 |
| english_name | 英文名称 |
| bounded_context | 所属限界上下文 |
| definition | 定义 |
| not_meaning | 不表示什么 |
| related_terms | 相关术语 |
| example | 示例 |
| owner | 术语负责人 |

## 6. 上下文映射查漏补缺

详细事件目录见 [50-上下文映射与领域事件目录](../06-接口与事件/50-上下文映射与领域事件目录.md)。

| 上游 | 下游 | 关系模式 | 协作方式 | 缺口 |
| --- | --- | --- | --- | --- |
| 主数据 | 所有业务系统 | 客户/供应方、发布语言 | 主数据启用/变更/停用事件 | 需要统一主数据事件版本和字段快照策略 |
| OMS | 中央库存 | 客户/供应方、开放主机服务 | 预占、释放、扣减请求；库存事件回传 | 需要明确预占失败、部分预占、超时释放策略 |
| OMS | WMS | 客户/供应方、防腐层 | 履约单下发，WMS 转出库单和任务 | 需要明确 OMS 出库指令与 WMS 出库单字段映射 |
| WMS | 中央库存 | 发布语言 | 上架完成、出库完成、盘点调整、冻结释放事件 | 需要统一 WMS 实物事实和库存入账的幂等键 |
| 采购 | WMS | 客户/供应方 | 采购到货计划/入库通知 | 需要明确采购订单、ASN、WMS 入库单三者关系 |
| 采购 | 供应商 | 伙伴关系/客户供应方 | PO 确认、ASN、退供协同 | 需要明确供应商回告修改交期/数量是否触发采购审批 |
| WMS/库存/OMS | BMS | 发布语言 | 作业事实、库存事实、履约事实 | 需要明确计费基准事件和重算/红冲策略 |
| 权限 | 所有系统 | 遵奉者 | 登录、Token、权限点、操作日志 | 需要统一数据权限维度：组织、仓库、货主、供应商 |

## 7. 领域事件命名规则

领域事件必须表示已经发生的业务事实，建议使用“名词 + 已 + 动作”的中文名称，英文代码使用过去式。

| 事件类型 | 中文名示例 | 英文代码示例 |
| --- | --- | --- |
| 主数据事件 | SKU已启用 | `SkuEnabled` |
| 单据事件 | 采购订单已审核 | `PurchaseOrderApproved` |
| 库存事件 | 库存已预占 | `InventoryReserved` |
| WMS 事件 | 出库单已发货 | `OutboundOrderShipped` |
| BMS 事件 | 费用明细已生成 | `FeeDetailGenerated` |

事件基础字段：

| 字段 | 说明 |
| --- | --- |
| event_id | 事件唯一 ID |
| event_type | 事件类型 |
| event_version | 事件版本 |
| occurred_at | 业务发生时间 |
| source_context | 来源上下文 |
| aggregate_type | 聚合类型 |
| aggregate_id | 聚合 ID |
| business_key | 业务单号 |
| idempotency_key | 幂等键 |
| trace_id | 链路 ID |
| payload | 事件业务载荷 |

## DDD 对齐说明

本文属于供应链全局领域地图层，负责说明核心域、支撑域、通用域和限界上下文边界。这里的“子系统”应理解为功能系统与限界上下文的候选映射，后续详细设计必须继续落到聚合、不变量、命令和领域事件。

## 10. 继续上下文

当前结论：供应链系统适合采用 DDD 进行“核心域重建模、支撑域适度建模、通用域复用”的设计策略。现有文档主流程完整，但需要补强领域地图、上下文映射、聚合不变量和统一事件目录。

关键假设：第一版系统以采购、OMS、中央库存、WMS、BMS、主数据、供应商、权限为核心，不立即展开计划补货、TMS、财务和报表分析的完整详细设计。

待决问题：是否要把售后从 OMS 中拆成独立售后上下文，取决于后续售后规则、退款、换货、补寄、维修、质检和财务联动复杂度。

下一步：建议继续补 `OMS-WMS 出库集成设计` 和 `库存履约异常与补偿设计`，因为这是销售出库链路里最容易出现跨系统不一致的部分。
