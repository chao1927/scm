# FE-REQ-004 WMS 与中央库存业务运维页面

## 1. 背景

WMS 后端已完成入库、收货、质检、上架、出库、波次、拣货、复核、交接、盘点、异常、Outbox/Inbox 等首轮能力；中央库存后端已完成库存账户、流水、预占、冻结、调整、快照、对账和事件投递。前端当前尚未提供仓储与库存运维页面。

## 2. 目标

- 提供 WMS Outbox/Inbox 失败事件查询、投递和重放入口。
- 提供中央库存余额、库存流水、快照、对账查询和关键操作入口。
- 严格使用后端已有真实读接口；WMS 作业单据若后端暂无列表读模型，本轮不伪造表格。

## 3. 范围

| 范围 | 说明 |
| --- | --- |
| WMS Outbox | 查询失败事件、人工重试、执行待投递事件扫描 |
| WMS Inbox | 查询失败入站事件、人工重放 |
| 库存余额 | 查询库存账户、展示在手/可用/预占/冻结 |
| 库存流水 | 查询库存台账 |
| 库存快照 | 查询快照、按库存账户生成快照 |
| 库存对账 | 查询对账单、确认待确认对账 |
| 库存 Outbox | 执行库存事件投递扫描 |

## 4. 不在本轮范围

- WMS 入库单、收货单、质检单、上架任务、出库单、波次、拣货、复核、交接、盘点的完整列表页。
- PDA 扫码模拟器。
- 库存冻结、调整、预占创建的复杂业务表单。
- 真实 WMS 与中央库存 RocketMQ 联调。

## 5. 验收标准

- 菜单“仓储作业”和“中央库存”可进入同一仓储与库存页面。
- 页面能调用 `/api/wms/v1/operations` 和 `/api/inventory/v1` 既有运维接口。
- WMS/库存投递、重试、重放、生成快照、确认对账成功后刷新列表。
- `npm run build`、后端 `mvn test -q`、`git diff --check` 通过。

## 6. 风险

- WMS 首轮后端缺少部分作业单据列表读模型，前端只能接运维可查范围。
- 页面中的库存快照和对账操作依赖已有库存账户数据。
- 权限 Web 测试仍需后续统一补齐。
