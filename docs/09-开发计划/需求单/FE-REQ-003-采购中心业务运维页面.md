# FE-REQ-003 采购中心业务运维页面

## 1. 背景

采购系统后端已完成请购、RFQ、比价、价格、采购订单、订单变更、到货跟踪、退供、供应商确认差异处理、Outbox/Inbox 和失败事件重放。前端当前仍未提供采购中心页面，运营人员无法在 Web 端查看采购主链路状态或处理常见异常。

## 2. 目标

- 打开采购中心菜单，提供采购主链路查询和关键状态推进能力。
- 覆盖请购、RFQ、采购订单、到货跟踪、供应商确认差异和失败事件重放。
- 将复杂新建单据、行级编辑、比价/价格维护留给后续更细需求单，避免一次性堆叠不可验收页面。

## 3. 范围

| 范围 | 说明 |
| --- | --- |
| 请购列表 | 按关键字、状态查询；支持草稿提交、审批中批准/拒绝 |
| RFQ 列表 | 按状态查询；支持草稿发布、已发布/报价中截标 |
| 采购订单列表 | 按状态查询；支持提交、审批、发布、取消、关闭 |
| 到货跟踪 | 查询 ASN、收货、质检、上架数量和异常状态 |
| 供应商确认 | 查询确认/差异事件；支持接受差异、重新协商、取消订单 |
| 失败事件 | 查询采购 Inbox 失败事件；支持人工重放 |

## 4. 不在本轮范围

- 请购、RFQ、PO 的复杂新建/编辑表单。
- 比价、价格、订单变更、退供的完整业务页面。
- 采购页面权限 E2E 自动化测试。
- 真实 WMS、供应商、BMS、TMS 远程 ACL 联调。

## 5. 验收标准

- 菜单“采购中心”可进入采购业务运维页面。
- 页面能调用 `/api/purchase/v1` 下既有查询和状态推进接口。
- 操作成功后刷新对应列表。
- `npm run build`、后端 `mvn test -q`、`git diff --check` 通过。

## 6. 风险

- 首轮页面以运营推进为主，不替代完整业务制单工作台。
- 部分状态动作使用默认运营原因，后续应在细化页面中改为专用表单。
- 后端 API 目前仍依赖本地默认权限上下文，权限 Web 测试需后续补齐。
