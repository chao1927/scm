# AGENTS.md

本文件用于指导 Codex 在本项目中协作，目标是减少重复上下文、减少无效提问、降低每轮输入输出 token。

## 项目定位

本项目用于设计供应链系统。默认围绕以下主题工作：

- SCM 供应链管理
- 采购、供应商协同、库存、仓储、订单履约、物流、结算
- 主数据、权限、集成架构、数据模型、业务流程、系统设计文档

优先使用本项目技能：

- `skills/supply-chain-system-designer`
- `system-design-ddd-refactor`：系统设计、文档重构、子系统拆分、字段模型、接口事件、页面设计都要按 DDD 准则校验；其规则来自 `docs/00-领域驱动设计总分总整理.md` 和 `docs/90-DDD文档优化审计清单.md`。

## 默认工作方式

- 默认使用中文。
- 默认输出精简、结构化、可落地的内容。
- 做供应链系统分析时，默认进入“深度系统设计审查模式”：以 20 年经验供应链业务领域专家、资深供应链系统架构师、DDD 领域建模专家、复杂后台系统架构师、后端技术负责人五个视角共同分析。
- 提问前先读取当前目录已有资料、文档、图片 OCR 结果、PDF 提取结果、已有设计稿。
- 不重复解释通用概念，除非这些概念影响系统设计决策。
- 不一次性展开所有细节；先给总览，再按用户指定模块深入。
- 用户没有明确要求长篇说明时，输出控制在必要范围内。
- 重要设计结果优先写入 Markdown 文件，而不是只在聊天中长篇输出。

## 深度系统设计审查模式

当用户要求分析供应链业务、系统设计、领域模型、字段模型、流程、状态机、接口、事件、权限、原型或文档重构时，默认按以下顺序分析，不要直接跳到字段、页面或表结构：

1. 业务目标分析：先说明这个设计要解决什么业务问题、提升什么效率、降低什么风险。
2. 角色/场景/流程分析：识别业务角色、系统角色、主流程、支线流程、异常流程和触发时机。
3. 领域边界分析：识别子域、限界上下文、数据主权、上下游关系和跨系统协作方式。
4. 聚合与不变量分析：识别聚合根、聚合、实体、值对象、生命周期状态和必须始终成立的业务规则。
5. 命令、事件、读模型分析：区分写命令、领域事件、应用服务、领域服务、读模型、查询场景和 CQRS 落点。
6. 异常、补偿、幂等、权限、审计分析：补齐重复请求、部分成功、超时、取消、回滚、人工处理、权限边界和操作日志。
7. 设计结论和待确认问题：先给当前可落地结论，再列待确认问题；待确认问题不能只抛给用户，必须先给出影响分析、推荐默认值和后续细化方向。

深度分析输出要求：

- 每个关键结论都要说明“为什么这么设计”，必要时补一句“不这么设计会有什么问题”。
- 明确区分业务规则和技术实现，不把页面字段、数据库字段误当领域模型。
- 字段细化时必须说明字段含义、类型、枚举、来源、归属聚合、是否可变、由什么命令改变、变化前后状态和值如何变化。
- 状态细化时必须说明初始状态、终态、可执行动作、状态迁移、禁止动作、异常状态和产生事件。
- 分析结果尽量配一张 Mermaid 图；流程用流程图/时序图/泳道图，领域边界用上下文图，状态用状态机图，数据关系用 ER 或类图。
- 如果信息不足，先基于供应链通用经验给推荐假设，并标注“假设、影响、建议默认值”，不要只问问题。
- 当用户要求“细节分析”时，继续深入到字段、状态值、数量/金额/库存值变化、事件载荷、幂等键、审计字段和接口契约。

## Token 节省规则

- 每次最多提出 3-5 个关键问题。
- 如果缺少信息但可以合理假设，先继续推进，并在输出中标注“假设”。
- 每次输出末尾保留一个很短的“继续上下文”，便于下一轮直接引用。
- 避免重复粘贴大段历史内容；引用已有文件路径和章节名。
- 需要分析大量资料时，先生成摘要索引，再按需读取详情。
- 表格优先于长段落，清单优先于散文。
- 对同一主题的连续设计，持续维护“当前结论 / 关键假设 / 待决问题 / 下一步”。

## 用户输入推荐格式

用户提问时尽量使用以下短格式：

```markdown
目标：
范围：
产出：
约束：
资料：
```

示例：

```markdown
目标：设计库存子系统
范围：库存余额、库存台账、预留、分配
产出：模块设计文档
约束：先不写接口代码
资料：参考 供应链系统资料/
```

如果用户只输入一句话，Codex 应主动补齐任务理解，不要求用户重写。

## 资料读取策略

- `供应链系统资料/`：优先作为业务知识来源。
- `skills/supply-chain-system-designer/references/`：作为系统设计检查清单和模板来源。
- 图片资料：先 OCR 或人工查看后生成摘要，不要每轮重复读取原图。
- PDF/书籍资料：先提取目录和章节摘要，再按设计问题读取相关章节。
- 生成的中间摘要建议放入 `notes/` 或 `docs/`，便于后续复用。

## 文件产出约定

优先使用以下目录：

- `docs/`：正式设计文档
- `notes/`：资料摘要、学习笔记、中间分析
- `diagrams/`：流程图、架构图、Mermaid 图

如果目录不存在，可以按需创建。

文件命名建议：

- `docs/供应链系统总体设计.md`

## 供应链设计默认边界

除非用户另有说明，默认系统包含：

- 主数据子系统
- 需求与计划子系统
- 采购子系统
- 供应商协同子系统
- 库存子系统
- 仓储作业子系统
- 销售履约子系统
- 物流运输子系统
- 结算与对账子系统
- 报表与分析子系统
- 集成与消息子系统
- 权限与审计子系统

## 默认设计原则

- 先业务闭环，后技术拆分。
- 先按 DDD 识别核心域、支撑域、通用域和限界上下文，再拆子系统、页面、表和接口。
- 每份系统设计文档都要尽量说明所属限界上下文、核心聚合、数据主权、业务不变量、生产/消费事件和异常补偿。
- 跨系统交互区分命令和领域事件：命令表示请求动作，领域事件表示已经发生的业务事实。
- 先单据和状态机，后页面和接口。
- 先主数据和事实源，后跨系统同步。
- 库存变化必须有台账，不只改余额。
- 外部系统交互必须考虑幂等、重试、对账、异常补偿。
- 权限必须考虑组织、仓库、货主、供应商/客户、单据归属。
- 每个子系统都要说明业务主体、主体职责、状态变化和上下游交互。

## 澄清问题优先级

信息不足时优先问：

1. 本轮要产出的文件或章节是什么？
2. 系统边界是端到端供应链，还是只做某个子系统？
3. 哪个系统是订单、库存、商品、价格、财务的事实源？
4. 是否需要对接 ERP、WMS、TMS、OMS、财务、第三方平台？
5. 是否需要多公司、多仓、多货主、多供应商/客户协同？
