# 工程编码工作流

## 核心目标

按“先理解、后设计、再编码、最后验证”推进。优先稳定接口、数据和状态规则，避免一开始写实现后反复返工。

## 编码前检查

除非用户明确要求直接编码，否则先完成以下内容；小任务可压缩，但不得跳过关键业务判断：

1. 用自己的话说明业务目标、输入、输出、关键规则和不做的范围。
2. 列出正常、异常和边界场景。
3. 明确字段、枚举、状态和数据来源；信息不够时列出待确认项及推荐默认假设。
4. 稳定方法签名、职责、输入输出和副作用。
5. 复杂流程画 Mermaid 流程图、状态机或时序图。
6. 给出至少正常、边界、异常三个测试样例并手工推演关键路径。

## 返工控制

- 先写领域规则、DTO/实体字段、枚举和方法签名，再接入 Service、Controller 和基础设施。
- 明确本次修改范围、保持稳定的公共接口和高风险假设。
- 不为小问题重构整个模块；不引入新框架，除非必要且与现有工程一致。
- 不删除用户已有逻辑，除非已说明兼容性风险并取得授权。
- 任务较大时按“模型和契约 -> 核心算法 -> 测试 -> 持久化/集成 -> 接口”分批，每批均可验证。

## 编码规范

- 保持类职责单一，方法短小，命名表达业务含义。
- 将复杂判断拆成可测试的私有方法；状态判断放入实体或枚举，避免散落魔法字符串。
- 为不直观的业务不变量写简短注释，不写重复代码含义的注释。
- Java/Spring Boot 默认遵循 `Controller -> Application Service -> Domain -> Repository/Mapper`；跨系统调用通过防腐层，已发生事实通过 Outbox/Inbox 事件。
- 优先使用已有框架、异常码、测试工具和项目模式。

## 流程图选择

- 多分支业务：`flowchart TD`。
- 生命周期或动作限制：`stateDiagram-v2`。
- 接口、数据库、消息队列协作：`sequenceDiagram`。

图必须使用业务语言、明确分支条件，并能解释代码结构。

## 验证与评审

每次代码改动后运行最相关的测试或编译；Java/Maven 项目优先：

```bash
mvn -pl <module> -am test
```

完成后检查：

| 检查项 | 要求 |
| --- | --- |
| 业务规则 | 正常、异常、边界均覆盖 |
| 状态与并发 | 状态迁移、乐观锁、幂等合理 |
| 空值与错误 | 输入校验、异常码、补偿路径明确 |
| 可测试性 | 关键规则有单元测试 |
| 修改范围 | 未引入无关重构或破坏公共接口 |
| 可读性 | 命名清晰、重复少、复杂逻辑已拆分 |

最终说明修改文件、验证命令及结果、仍存风险。若未完成，明确未完成项，不得宣称完成。
