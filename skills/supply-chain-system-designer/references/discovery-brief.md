# 需求发现简报

项目处于早期阶段或用户目标比较宽泛时使用本参考。需求发现要短、聚焦决策。

## 快速上下文快照

记录这些字段：

| 字段 | 记录内容 |
| --- | --- |
| 业务目标 | 降本、提升交付准确率、库存可视化、供应商协同、合规、计划准确率、运营提效 |
| 范围 | 采购、库存、仓储、履约、物流、计划、财务结算、质量、供应商/客户门户 |
| 公司模式 | 单公司、集团公司、多租户 SaaS、多货主仓、跨境业务 |
| 用户 | 采购员、计划员、仓库操作员、物流协调员、供应商、客服、财务、管理者、管理员 |
| 系统 | ERP、WMS、TMS、MES、OMS、SRM、CRM、财务、BI、主数据、电商平台、3PL |
| 规模 | 每日订单行数、SKU 数、仓库数、合作伙伴数、API QPS、批处理规模 |
| 约束 | 存量系统、合规、预算、云/本地部署、上线节奏、报表要求 |

## 最小提问策略

只问会改变架构或领域建模的问题：

1. 第一个必须端到端跑通的运营流程是什么？
2. 商品、库存、订单、价格、合作伙伴数据分别由哪个系统作为事实源？
3. 哪些单据会产生财务义务？
4. 库存和仓库作业由本系统承载，还是从 WMS 集成？
5. 当前最耗费人工的异常是什么？

## 默认假设

用户希望快速推进时使用这些假设：

- 系统支持多供应商、多仓库、多 SKU 和多业务组织。
- ERP 或财务系统仍是会计记录和最终结算的事实源。
- 新系统负责流程编排、可视化、协同和运营异常处理。
- 外部系统调用必须幂等且可审计。
- 库存变化需要台账式记录，不能只维护可变余额。
