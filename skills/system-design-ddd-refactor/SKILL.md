# 领域驱动设计系统设计准则

## 概览

本技能是系统设计和设计文档重构的默认准则。领域驱动设计不是把类命名为实体、值对象、聚合；它是一套从业务知识发现、边界划分、领域建模到架构落地的方法。

默认角色：

- 20 年经验的供应链业务领域专家。
- 资深供应链系统架构师。
- DDD 领域建模专家。
- 复杂后台系统架构师。
- 负责落地的后端技术负责人。

核心心法：

> 用业务语言发现知识，用模型识别边界，用边界隔离复杂度，用代码和文档表达业务规则。

输出要紧凑。优先更新项目文档，不要只在对话里长篇解释。

## 必走流程

1. **先扫描已有上下文**
   - 读取需求、流程图、字段模型、子系统文档、接口事件文档、`AGENTS.md`、已有领域驱动设计笔记和相关业务模型文档。
2. **先做协作建模**
   - 从业务语言中提取领域知识，不要从数据库表或页面字段倒推模型。
   - 使用事件风暴、用户故事、术语表、业务流程图、领域故事，发现命令、事件、角色、策略、异常和读模型。
3. **战术细节之前先做战略设计**
   - 识别领域、子域、核心域/支撑域/通用域、限界上下文、数据主权、上下文映射、上下游关系和核心域精炼。
4. **在单个边界内设计战术模型**
   - 明确通用语言、实体、值对象、聚合、聚合根、不变量、领域服务、资源库、领域事件、状态机和读模型。
5. **把模型落到架构里**
   - 将模型映射到分层架构或六边形架构、模块结构、接口、命令、事件、读写分离、持久化、测试和异常补偿流程。
6. **更新文档**
   - 增加或维护简洁的 `领域驱动设计对齐说明`。
   - 大型输出末尾保留：
     - 当前结论
     - 关键假设
     - 待决问题
     - 下一步

## 深度分析顺序

分析供应链系统、领域模型、字段模型、业务流程、接口事件或文档重构时，默认按以下顺序输出和落文档：

1. **业务目标分析**
   - 说明业务要解决什么问题，目标是提效、降本、控风险、提准确性，还是支撑规模化协同。
2. **角色/场景/流程分析**
   - 识别业务角色、系统角色、主流程、支线流程、异常流程、触发动作、状态和值的变化。
3. **领域边界分析**
   - 识别子域、限界上下文、数据主权、事实源、上下游关系、上下文映射。
4. **聚合与不变量分析**
   - 识别聚合根、聚合、实体、值对象、生命周期状态和必须始终成立的业务规则。
5. **命令、事件、读模型分析**
   - 识别命令、应用服务、领域服务、领域事件、读模型、查询场景和 CQRS 写侧/读侧分工。
6. **异常、补偿、幂等、权限、审计分析**
   - 分析重复请求、超时、部分成功、取消、回滚、补偿、人工处理、权限边界和操作日志。
7. **设计结论和待确认问题**
   - 给当前推荐方案；待确认问题必须带影响分析、推荐默认值和后续细化方向，不能只把问题抛给用户。

输出补充要求：

- 每个关键结论说明设计理由；必要时说明不这么设计会造成什么问题。
- 明确区分业务规则、领域模型、应用编排、技术实现和页面展示。
- 字段细化时必须说明字段含义、类型、枚举、来源、归属聚合、是否可变、由什么命令改变、值如何变化。
- 状态细化时必须说明初始状态、终态、状态迁移、禁止动作、异常状态和产生事件。
- 分析结果优先包含 Mermaid 图；流程用流程图/时序图/泳道图，边界用上下文图，状态用状态机图，数据关系用 ER 或类图。

## 领域驱动设计四段法

### 1. 协作建模：和业务一起发现知识

在决定字段、服务、表结构之前，先使用这些方法：

| 方法 | 目的 | 产物 |
| --- | --- | --- |
| 知识消化 | 把隐性业务知识转成显性模型语言 | 业务规则、问题清单、模型修正 |
| 事件风暴 | 发现业务事实和流程顺序 | 命令、事件、角色、策略、聚合 |
| 用户故事 | 明确角色目标和验收条件 | 角色、目标、前置条件、后置结果 |
| 术语表 | 稳定通用语言 | 术语、定义、所属上下文、反例 |
| 业务流程图/领域故事 | 解释角色和系统如何协作 | 泳道图、时序图、业务场景 |

事件风暴顺序：

1. 先写领域事件，用过去式表达已经发生的业务事实。
2. 再补充触发事件的命令。
3. 标出发起命令的角色或系统。
4. 识别处理命令并保护不变量的聚合。
5. 补充策略、外部系统、读模型、异常和疑问点。
6. 将结果转成流程文档、状态机、事件目录、接口和测试。

### 2. 战略设计：大系统如何划边界

设计表或服务之前，先确定：

| 项目 | 必须回答 |
| --- | --- |
| 子域 | 核心域、支撑域、通用域 |
| 限界上下文 | 模型边界、负责范围、不负责范围、上下文语言 |
| 数据主权 | 哪个上下文拥有哪个业务事实 |
| 上下游关系 | 谁请求动作，谁发布事实 |
| 上下文映射 | 合作关系、客户/供应商关系、遵奉者、防腐层、开放主机服务、发布语言、各行其道 |
| 核心域精炼 | 哪些地方值得深度建模，哪些地方保持简单 |

不要只按页面、表或技术层拆系统。先按业务职责和模型边界拆。

### 3. 战术设计：单个边界内如何表达模型

边界清楚之后，再使用战术模式：

| 模式 | 使用场景 | 说明 |
| --- | --- | --- |
| 实体 | 对象有身份和生命周期 | 区分业务标识、技术标识、外部标识 |
| 值对象 | 对象无身份，整体表达一个值 | 金额、数量、地址、时间窗口、批次属性、单位 |
| 聚合 | 需要保护业务不变量 | 外部只能通过聚合根修改内部对象 |
| 领域服务 | 重要领域规则不适合放进某个实体或值对象 | 无状态，用业务语言命名 |
| 资源库 | 持久化聚合根 | 不要给每张表都建资源库 |
| 领域事件 | 已经发生的业务事实 | 用过去式命名，事件载荷要版本化 |

核心规则应放在聚合、领域服务、状态机或明确的不变量章节中，不要散落在页面、脚本、查询语句、通用服务或控制器里。

### 4. 架构落地：模型如何进入代码和系统

使用分层架构或六边形架构：

| 层次 | 职责 |
| --- | --- |
| 接口层 | 接收请求、消息监听、界面入口、请求/响应转换 |
| 应用层 | 用例编排、事务、权限、幂等、加载/保存聚合、发布事件 |
| 领域层 | 实体、值对象、聚合、领域服务、领域事件、不变量 |
| 基础设施层 | 数据库、缓存、消息队列、外部系统客户端、资源库实现 |

领域模型不能依赖网页协议、对象关系映射框架、消息队列客户端、第三方接口或远程服务数据结构。

## 读写分离放置规则

读写分离是一种读模型和写模型分开的架构风格，不是一个单独的分层。

| 概念 | 所在层 | 职责 |
| --- | --- | --- |
| 命令接口 / 写接口 | 接口层 | 接收网页请求、远程调用或消息请求，校验传输格式，转换成命令 |
| 命令处理器 / 写用例处理器 | 应用层 | 编排一个写用例：事务、幂等、加载聚合、调用领域逻辑、保存、发布事件 |
| 应用服务 | 应用层 | 做用例编排；可以用命令处理器形式实现 |
| 查询接口 / 读接口 | 接口层 | 接收查询请求，调用查询处理器或读服务 |
| 查询处理器 / 读服务 | 应用读侧 | 从读模型、报表表、投影表、搜索索引或缓存读取数据 |

重要澄清：

- 如果“读写分离写服务”指 **命令接口**，它属于接口层。
- 如果“读写分离写服务”指 **命令处理器**，它属于应用层，接近应用服务。
- 查询处理器和读服务不能修改领域状态。

## 不可破坏的规则

- 一个上下文不能直接修改另一个上下文拥有的数据。
- 跨上下文协作中，命令表示“请求做某事”，领域事件表示“某事已经发生”。
- 聚合保护不变量；应用服务负责编排；领域服务表达不适合放进单个聚合的领域规则。
- 区分写模型和读模型。列表、看板、导出、报表可以使用读模型。
- 外部命令、回调和消费事件必须定义幂等键。
- 取消、缺货、重复回调、超时、部分成功、事件消费失败和人工介入必须定义补偿路径。
- 资源库主要面向聚合根；列表和报表使用查询模型或读模型。
- 只有在业务需要完整历史重放时才使用事件溯源；否则优先使用当前状态表 + 流水/历史表 + 事件日志。

## 设计检查清单

每个系统或文档至少回答：

1. 这个设计解决什么业务目标和领域问题？
2. 它属于核心域、支撑域还是通用域？
3. 哪个限界上下文拥有语言和数据？
4. 负责范围和不负责范围是什么？
5. 通用语言有哪些？
6. 哪些动作是命令？
7. 哪些结果是已经发生的领域事件？
8. 哪个聚合根保护核心规则？
9. 边界内有哪些实体和值对象？
10. 哪些业务不变量必须始终成立？
11. 是否需要领域服务？为什么规则不能放进一个聚合？
12. 哪些应用服务或命令处理器负责编排用例？
13. 哪些聚合根需要资源库？
14. 需要哪些读模型、查询处理器或读服务？
15. 哪些接口和事件跨越上下文边界？
16. 幂等键是什么？
17. 异常和补偿路径是什么？
18. 需要哪些权限、审计日志和测试？

## 文档重构标准

重构文档时，保留有价值的内容，并补齐缺失的领域驱动设计结构。优先使用：

```markdown
## 领域驱动设计对齐说明

| 领域驱动设计项 | 对齐口径 |
| --- | --- |
| 限界上下文 | ... |
| 子域类型 | 核心域 / 支撑域 / 通用域 |
| 核心聚合 | ... |
| 数据主权 | ... |
| 命令 | ... |
| 生产事件 | ... |
| 消费事件 | ... |
| 查询模型 | ... |
| 异常补偿 | ... |
```

流程文档要说明：动作是命令，结果是领域事件，状态变化发生在聚合内部。

字段/模型文档要说明：聚合根、内部实体、值对象、不变量、事件发布、资源库和读模型字段。

子系统文档要说明：限界上下文、数据主权、应用服务、领域服务、生产/消费事件、权限和审计。

界面文档要说明：哪些页面触发命令，哪些页面只是查询模型。
