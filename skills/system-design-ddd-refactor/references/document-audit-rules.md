# 设计文档审计规则

## 1. 审计目标

审计或重构系统设计文档时，要确保每份文档都能放回完整的领域驱动设计体系中使用，而不是只作为孤立笔记存在。

审计重点不是把文档写长，而是让文档回答清楚：

1. 这份文档属于哪个设计层级。
2. 它描述哪个限界上下文，或哪些上下文之间的协作。
3. 它涉及哪些聚合、实体、值对象、命令、事件和不变量。
4. 它的状态变化、数据主权、查询模型和异常补偿是否清楚。
5. 它和其他文档之间是否能互相衔接。

## 2. 最小对齐要求

每份设计文档应有简洁的 `领域驱动设计对齐说明` 或等价章节，至少识别这些内容：

1. 文档层级：业务地图、业务流程、领域模型、主数据、子系统、接口事件、界面、审计。
2. 限界上下文：当前文档所属上下文，或跨上下文范围。
3. 领域对象：核心聚合、实体、值对象。
4. 数据主权：哪个上下文拥有数据，哪个上下文只读、缓存或引用。
5. 命令动作：哪些业务动作会改变状态。
6. 生产事件：状态变化后发布哪些领域事件。
7. 消费事件：依赖哪些上游事实来改变本地数据。
8. 业务不变量：必须始终成立的规则。
9. 幂等与补偿：重复请求、失败、取消、回滚如何处理。
10. 查询边界：哪些内容属于读模型、列表、报表、看板或导出。

## 3. 按文档类型审计

### 3.1 业务总览文档

业务总览文档要补充：

1. 核心域、支撑域、通用域划分。
2. 主要限界上下文和上下游依赖。
3. 数据主权总览。
4. 核心业务闭环。
5. 关键风险点和后续深化顺序。

### 3.2 业务流程文档

每个关键步骤要能回答：

1. 哪个角色或系统发起命令。
2. 哪个限界上下文处理命令。
3. 哪个聚合发生状态变化。
4. 状态如何从旧状态变成新状态。
5. 产生哪个领域事件。
6. 下一个上下文如何消费事件。
7. 失败、取消、超时、重复提交如何补偿。

### 3.3 字段模型文档

字段文档不能只是表字段清单，应补充：

1. 聚合根和内部实体。
2. 值对象。
3. 状态字段和状态迁移规则。
4. 业务不变量。
5. 字段可变性和版本规则。
6. 事件字段和快照字段。
7. 只属于查询模型的冗余字段。

### 3.4 主数据文档

主数据应被视为上游发布语言，要说明：

1. 权威来源。
2. 版本规则。
3. 启用、变更、停用事件。
4. 下游系统如何订阅或同步。
5. 下游使用时是否需要历史快照。
6. 外部来源数据是否需要防腐层转换。

### 3.5 子系统设计文档

子系统文档应补充限界上下文表：

| 项目 | 必填内容 |
| --- | --- |
| 限界上下文 | 上下文名称和职责边界 |
| 不负责范围 | 明确排除的职责 |
| 核心聚合 | 本上下文负责维护的聚合 |
| 数据主权 | 本上下文拥有的数据 |
| 生产事件 | 本上下文发布的业务事实 |
| 消费事件 | 本上下文依赖的外部业务事实 |
| 查询模型 | 列表、详情、看板、导出 |
| 权限审计 | 角色、数据范围、操作日志 |

### 3.6 界面设计文档

界面应区分三类页面：

1. 命令页面：新增、编辑、审核、取消、释放、扣减、发货、收货。
2. 查询页面：列表、详情、看板、日志、追踪。
3. 配置页面：枚举、策略、规则、参数。

每个行级操作都应能映射到一个命令或查询，不能只停留在按钮名称。

### 3.7 接口与事件文档

每个命令或事件必须说明：

1. 来源上下文。
2. 目标上下文。
3. 涉及聚合。
4. 载荷字段。
5. 幂等键。
6. 版本号。
7. 重试规则。
8. 失败补偿。
9. 是否需要消息顺序。
10. 是否影响查询模型刷新。

## 4. 审计状态

| 状态 | 含义 |
| --- | --- |
| 已对齐 | 已有领域驱动设计说明，且关键概念覆盖完整 |
| 待深化 | 已有对齐说明，但接口、聚合、规则还需要细化 |
| 缺失 | 缺少必要的边界、模型或事件内容 |
| 延后 | 明确不在当前阶段处理，后续再设计 |

## 5. 高优先级缺口

审计时优先跟踪这些高风险缺口：

1. 订单系统和仓储系统的出库集成。
2. 库存预占失败、扣减失败、释放失败和补偿。
3. 主数据事件版本和字段快照。
4. 计费事件、对账和结算边界。
5. 聚合与字段稳定后的数据库设计。
6. 按组织、仓库、货主、供应商、客户、单据归属划分权限。

## 6. 维护规则

每次修改设计文档后，应在审计文档或本次总结中记录：

1. 修改了哪些文件。
2. 新增了哪些领域驱动设计对齐内容。
3. 还剩哪些高优先级缺口。
4. 下一步最建议深化哪个设计切片。
