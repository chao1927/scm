# 领域驱动设计系统设计规则

## 1. 核心定位

领域驱动设计是一套处理业务复杂度的方法。目标不是把类命名为实体或聚合，而是让业务专家、产品、架构师和开发人员围绕同一套模型协作，并让模型进入文档、接口、事件、状态、字段、表结构、测试和页面。

核心心法：

> 用业务语言发现知识，用模型识别边界，用边界隔离复杂度，用代码和文档表达业务规则。

对供应链系统来说，库存可用性、履约分配、仓内作业、调拨在途、采购入库、售后退货质检、供应商退货结算，都是需要重点建模的领域。

## 2. 协作建模

领域驱动设计从发现知识开始。不要从数据库表或页面字段开始。

### 2.1 知识消化

把业务表述转成显性模型事实：

1. 业务人员描述业务现象。
2. 设计人员追问词语在当前上下文中的含义。
3. 业务人员补充规则、异常和责任边界。
4. 设计人员形成模型术语。
5. 双方验证边界场景。
6. 模型进入流程、状态、事件、接口、字段和测试文档。

持续追问：

- 谁发起这个动作？
- 为什么会发生？
- 发生前必须满足什么条件？
- 哪些状态、数量、金额、地点、归属或义务会变化？
- 会产生什么事件？
- 谁需要知道这件事已经发生？
- 如果失败、重复、超时或部分成功，如何处理？

### 2.2 通用语言

每个限界上下文拥有自己的语言。同一个词在不同上下文中可以有不同含义。

例如“库存”：

| 上下文 | 含义 |
| --- | --- |
| 订单上下文 | 是否可售、是否可履约 |
| 中央库存上下文 | 可用、预占、冻结、在途、不可售 |
| 仓储上下文 | 仓库、库区、库位、批次、容器上的实物数量 |
| 计费上下文 | 可计费、可对账、可结算的库存数量 |

应维护术语表、状态词典、动作表、事件目录和模型词典。

### 2.3 事件风暴

事件风暴用于发现业务流程和模型边界。

1. 领域事件：已经发生的业务事实，用过去式表达。
2. 命令：触发事件的请求动作。
3. 角色/系统：用户、外部系统、定时任务。
4. 聚合：处理命令并保护不变量的对象。
5. 策略：事件发生后的自动反应。
6. 读模型：查询、展示、投影需求。
7. 异常/疑问：业务缺口和补偿路径。

供应链示例：

| 元素 | 示例 |
| --- | --- |
| 命令 | 审核采购订单、预占库存、确认出库 |
| 事件 | 采购订单已审核、库存已预占、出库单已发货 |
| 角色 | 采购员、仓库作业员、订单系统、仓储系统 |
| 聚合 | 采购订单、库存预占、出库单 |
| 策略 | 订单已支付后自动预占库存 |
| 读模型 | 库存看板、履约追踪页 |

### 2.4 用户故事和领域故事

用户故事格式：

```text
作为 <角色>
我希望 <完成某个业务动作>
以便 <达成业务目标>
```

还要补充前置状态、命令、业务规则、状态变化、产生事件、异常分支、权限和审计要求。

领域故事描述角色之间如何协作，可进一步转成泳道图或时序图。

## 3. 战略设计

战略设计决定大系统如何切分。

### 3.1 领域和子域

| 类型 | 含义 | 设计策略 |
| --- | --- | --- |
| 核心域 | 体现业务差异、规则复杂、对业务成败关键 | 深入建模，明确聚合、不变量和事件 |
| 支撑域 | 重要但不构成核心差异 | 适度建模，保持稳定集成 |
| 通用域 | 通用能力 | 复用成熟模型，避免过度设计 |

判断核心域的问题：

- 这个能力是否创造业务优势？
- 它是否经常变化？
- 它是否包含大量规则和异常？
- 建模不好是否会影响成本、收入、履约或客户体验？
- 它是否难以直接购买成熟产品替代？

### 3.2 限界上下文

每个上下文都要定义：

- 业务目标
- 负责范围和不负责范围
- 通用语言
- 拥有的数据
- 核心聚合
- 上游/下游上下文
- 接受的命令
- 生产的事件
- 消费的事件
- 一致性要求
- 风险和待决问题

### 3.3 上下文映射

| 模式 | 含义 |
| --- | --- |
| 合作关系 | 两个上下文共同演进 |
| 客户/供应商关系 | 上游向下游提供能力或模型 |
| 遵奉者 | 下游接受上游模型 |
| 防腐层 | 将外部模型转换成本地模型 |
| 开放主机服务 | 为多个消费者提供稳定接口或事件协议 |
| 发布语言 | 共享的消息或数据格式 |
| 各行其道 | 集成成本不值得，选择不集成 |
| 大泥球隔离 | 把遗留混乱模型隔离在转换层之后 |

## 4. 战术设计

### 4.1 实体

当对象有身份和生命周期时使用实体。先设计身份，再设计字段。区分业务标识、技术标识和外部标识。

### 4.2 值对象

当对象没有身份、需要整体校验时使用值对象，例如金额、数量、地址、联系人、时间窗口、批次属性、单位、评分。

值对象应尽量不可变，并在创建时完成校验。

### 4.3 聚合

聚合用于保护不变量。

规则：

- 聚合边界按不变量划分，不按外键划分。
- 外部对象只能通过聚合根修改聚合内部。
- 一个事务通常只修改一个聚合。
- 引用其他聚合时使用标识，不直接持有内部对象。
- 跨聚合一致性通过领域事件完成。
- 聚合要足够小，避免并发冲突过大。

### 4.4 领域服务

领域服务用于表达重要领域行为，但该行为不自然属于某一个实体或值对象。领域服务应无状态，并用业务语言命名。

示例：

- 库存分配领域服务
- 履约拆单领域服务
- 供应商评分计算领域服务

不要把领域服务当成应用编排逻辑的垃圾桶。

### 4.5 应用服务

应用服务用于用例编排：

- 权限校验
- 事务边界
- 幂等控制
- 加载聚合
- 调用聚合或领域服务
- 保存聚合
- 发布事件
- 必要时调用外部端口

核心业务规则不应放在应用服务里。

### 4.6 资源库

资源库主要面向聚合根。资源库表达对聚合集合的访问，不是通用表网关。

列表、报表、看板、导出、搜索应使用查询模型或读模型。

### 4.7 领域事件

领域事件是已经发生的业务事实。名称使用过去式。

事件字段建议：

- 事件标识
- 事件类型
- 事件版本
- 发生时间
- 来源上下文
- 聚合类型
- 聚合标识
- 业务主键
- 幂等键
- 链路追踪标识
- 事件载荷

## 5. 架构落地

优先使用分层架构或六边形架构。

| 层次 | 职责 |
| --- | --- |
| 接口层 | 接口、消息监听、界面入口、请求/响应数据结构 |
| 应用层 | 用例编排、事务、权限、幂等、事件发布 |
| 领域层 | 实体、值对象、聚合、领域服务、领域事件、不变量 |
| 基础设施层 | 数据库、缓存、消息队列、外部系统客户端、资源库实现 |

领域模型不应依赖对象关系映射框架、网页协议工具、消息队列客户端或第三方模型。

### 5.1 六边形端口

使用端口隔离领域/应用与基础设施：

- 资源库端口
- 事件发布端口
- 外部服务端口
- 时钟/标识生成端口
- 事务/幂等端口

### 5.2 读写分离

读写分离用于分离写模型和读模型。它本身不是一层。

| 概念 | 所在层 | 职责 |
| --- | --- | --- |
| 命令接口 | 接口层 | 接收写请求并创建命令 |
| 命令处理器 | 应用层 | 编排写用例 |
| 应用服务 | 应用层 | 用例编排，可由命令处理器实现 |
| 查询接口 | 接口层 | 接收查询请求 |
| 查询处理器 / 读服务 | 应用读侧 | 查询读模型、投影或报表 |

如果“读写分离写服务”指命令接口，它属于接口层；如果指命令处理器，它属于应用层。

### 5.3 事件、长流程和一致性

跨上下文事实使用领域事件。使用事件发布表和事件收件箱保证可靠发布和消费。长流程使用流程管理器处理重试、超时、补偿和人工介入。

只有当业务需要完整历史事件流重放时，才使用事件溯源。否则优先使用当前状态表 + 流水/历史表 + 事件日志。
